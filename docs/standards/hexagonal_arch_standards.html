Application vs Domain: A camada application/ orquestra o domínio. Os Services implementam os in-ports e coordenam domain models + out-ports. Os Commands são Java records imutáveis que carregam dados de entrada. Os Ports definem contratos com o mundo externo (in = quem chama, out = quem é chamado). Pode importar domain, mas nunca importa adapter.
📌 Nota importante: A application/ contém apenas:
  • Ports (interfaces) — port/in/ (in-ports) e port/out/ (out-ports)
  • Commands — command/ (DTOs de entrada para use cases)
  • Results — result/ (DTOs de saída de use cases e out-ports)
As implementações de Services/UseCases ficam em adapter/in/ como POJOs plain (sem @Service). Exemplo: adapter/in/trail/usecase/ExploreTrailsUseCase.java implementa application/trail/port/in/ExploreTrailsUsecase.java

Fluxo Request → Command → Domain → Response: O Controller recebe CreateExpeditionRequest (DTO HTTP com @NotBlank), o WebMapper converte para CreateExpeditionCommand (application), passa para o use case, e na volta mapeia Domain → ExpeditionResponse.
📋 Regra de DTOs na camada HTTP:
  • Request DTOs — decorações HTTP (@NotBlank, @Valid, @JsonProperty) — ficam em adapter/in/web/[feature]/dto/
  • Response DTOs — serialização HTTP — ficam em adapter/in/web/[feature]/dto/
  • Mapeamento simples (1:1 sem dependências) — método na própria classe (Request.toCommand(), Response.fromDomain())
  • Mapeamento complexo (com injeção de dependências) — classe dedicada em adapter/in/web/[feature]/mapper/[Feature]WebMapper.java

🎯 Mappers por camada:
  • WebMapper (adapter/in/web/[feature]/mapper) — Request/Response ↔ Command/Domain
  • PersistenceMapper (adapter/out/persistence/[feature]/mapper) — Domain ↔ JPA Entity
  • External API Mappers (adapter/out/[service]/mapper) — Wire DTO → Boundary object (ex: OverpassElement → OverpassTrailData)
  • Technical Mappers (adapter/out/persistence/[feature]/mapper) — Ex: LineStringMapper (GeoPoint → JTS LineString). Nunca na application layer!
Regra geral: Mappers vivem no adapter que "conhece" ambos os tipos. Domain não mapeiam nada. Application não mapeiam tipos de infraestrutura (JTS, JPA, etc).
📋 DTOs externos (External APIs):
  • Wire DTOs (JSON deserialization) — adapter/out/[service]/ com annotations Jackson (@JsonIgnoreProperties, etc)
  • Boundary objects — application/trail/result/ (ex: OverpassTrailData) — application vê isto, nunca vê o wire DTO
  • Mapeamento — Dentro do Adapter (ex: OverpassClient) antes de cruzar a porta. Não tem mapper classe separada se for lógica simples.

// Services são POJOs — wiring manual via @Bean
@Bean
ExpeditionService expeditionService(
ExpeditionRepository repo,
ExpeditionEventPublisher events ) {
return new ExpeditionService(repo, events); }
Por que wiring manual? Os Services no application/ são POJOs sem @Service — mantendo a camada livre de Spring. O BeanConfig é o único lugar que conhece tanto os ports quanto as implementações concretas, conectando tudo via construtor.
